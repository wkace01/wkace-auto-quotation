# 바이브 코딩 회고 — 잘한 점 & 배울 점
### 우경정보통신 스마트 견적서 자동화 프로젝트

> 작성일: 2026-03-24
> 목적: 팀 내 바이브 코딩 노하우 공유
> 대상: 비전공자 / 업무 자동화에 관심 있는 동료

---

## 들어가며

이 문서는 JavaScript를 몰랐던 상태에서 AI(Claude)와 함께 풀스택 웹 애플리케이션을 만든 경험을 분석한 회고입니다. 잘했던 것과 아쉬웠던 것을 솔직하게 정리했습니다. 비슷한 시도를 하려는 동료들에게 실질적인 참고가 되길 바랍니다.

---

## 1. 잘했던 점

### ✅ 1-1. 요구사항을 "왜"와 함께 전달했다

단순히 "이렇게 해줘"가 아니라 **문제의 원인과 원하는 결과를 같이 설명**했습니다.

> **나쁜 예**: "고객명 수정해줘"
> **잘한 예**: "고객명이 길어질 경우 가운데 정렬이 안돼. 그래서 a10 하나의 셀로 병합해뒀어. 해당 셀로 매핑하도록 수정해줘. 단, 앞에 '고객명 : ' 텍스트도 같이 들어가도록 해줘."

이렇게 전달하면 AI가 **의도를 추측하지 않아도 되기 때문에** 오류가 줄어듭니다. 실제로 이 요구사항은 한 번에 정확히 구현되었습니다.

---

### ✅ 1-2. 원하지 않는 방향의 계획은 즉시 거절하고 재지시했다

AI가 제안한 구현 방향이 내가 원하는 것과 다를 때 **명확하게 거절하고 올바른 방향을 다시 설명**했습니다.

> **상황**: AI가 "읽기 전용 '조정 후 금액' 행 추가"를 제안
> **반응**: "읽기 전용 행을 추가하지마. 현재 ± 버튼으로 숫자 수정할 수 있는 거기에 있는 숫자를 할인율이나 조정 배수 버튼으로 조정하고 싶어."

계획(Plan) 단계에서 이를 잡아냈기 때문에 **잘못된 코드가 작성되기 전에** 방향을 바로잡았습니다. 구현 후 되돌리는 것보다 계획 단계 거절이 훨씬 효율적입니다.

---

### ✅ 1-3. 완료된 작업은 계획에서 즉시 삭제하며 관리했다

> "서버 PDF 페이지 분할 버그 수정 해결 됐어 계획에서 삭제해줘"

진행 중인 작업 목록을 실시간으로 정리했습니다. 이렇게 하면 AI가 이미 해결된 문제를 다시 건드리는 혼선을 방지할 수 있습니다. **작업 컨텍스트를 깔끔하게 유지하는 습관**입니다.

---

### ✅ 1-4. 기능 단위로 검증하고 다음 단계로 넘어갔다

기능 하나를 완성하면 반드시 **로컬에서 직접 테스트**한 뒤 다음 기능으로 넘어갔습니다.

- 배수 기능 구현 → 테스트 → "실시간으로 변경이 안되는데?" → 재디버깅 요청
- 카카오 주소 검색 수정 → "제대로 수정했네 잘했어" 확인 → 다음 이슈로 이동

기능을 쌓아가면서 검증하지 않으면 나중에 어디서 문제가 생겼는지 찾기가 매우 어렵습니다. **작게 만들고 바로 확인하는 루틴**이 몸에 배어 있었습니다.

---

### ✅ 1-5. 커밋과 PR 흐름을 프로젝트 초반부터 적용했다

Git을 처음 다루면서도 커밋 → 브랜치 → PR 흐름을 세션 전반에 걸쳐 일관되게 유지했습니다. 덕분에 **작업 이력이 남아** 어느 시점에 무엇이 바뀌었는지 추적이 가능합니다. 코드를 되돌려야 할 상황이 생겨도 특정 커밋으로 돌아갈 수 있는 안전망이 됩니다.

---

### ✅ 1-6. 도메인 지식을 정확하게 전달했다

소방설비 유지관리 업무의 견적 계산 구조(성능점검·유지점검·위탁선임, 등급 기준, 노임단가 적용 방식)를 AI에게 명확하게 전달할 수 있었습니다. **업무를 아는 사람만이 줄 수 있는 맥락** 덕분에 AI가 단순 코딩 도구 이상의 역할을 할 수 있었습니다.

## 2. 아쉬웠던 점 & 개선 방법

### ⚠️ 2-1. 버그가 어느 파일에서 발생했는지 먼저 좁혀주면 좋았다

**무슨 일이 있었나**
카카오 주소 검색이 갑자기 안 되는 버그가 발생했습니다. 실제 원인은 AI가 수정해야 할 파일(`index.html`)이 아닌 다른 파일(`toss_step_index.html`)을 잘못 수정했기 때문이었습니다. 서버는 `index.html`만 사용하는데 AI가 그것을 혼동한 것입니다.

**왜 늦게 발견됐나**
"카카오 안됨 → 디버깅해줘" 요청만 했고, 어떤 파일이 실제로 서버에서 동작하는지 서로 확인하지 않았습니다.

**개선 방법**
기능이 갑자기 안 될 때, "어떤 파일이 지금 실제로 실행되고 있어?"를 먼저 확인하는 습관. 증상보다 **범위를 먼저 좁히는 질문**이 효과적입니다.

> "카카오 검색이 안돼. 어제 어떤 파일들을 수정했어? 그 중에 카카오 관련된 게 있어?"

---

### ⚠️ 2-2. 서버 재시작을 반복 요청하기 전, 코드가 실제로 반영됐는지 먼저 확인

**무슨 일이 있었나**
코드 수정 후 기능이 안 된다고 "호스팅을 끄고 다시 시작해줘"를 두 번 요청했습니다. 그러나 실제 원인은 서버 자체가 아니라 코드 오류였습니다.

**왜 이렇게 됐나**
로컬에서 수정된 코드가 브라우저에 캐시로 남아 있거나, 코드 자체에 오류가 있어도 증상이 비슷하게 보일 수 있습니다. 서버 재시작은 임시방편일 뿐입니다.

**개선 방법**
기능이 안 될 때 순서대로 확인:
1. 브라우저 강제 새로고침 (`Ctrl + Shift + R`)
2. 브라우저 콘솔(F12 → Console 탭)에 빨간 에러가 있는지 확인
3. 그래도 안 되면 서버 재시작 요청

> "브라우저 콘솔에 에러 있어?" 라고 AI에게 먼저 물어보는 것도 좋습니다.

---

### ⚠️ 2-3. "안돼"보다 "이렇게 했을 때 이런 결과가 나와"로 설명하면 더 빠르다

**무슨 일이 있었나**
배수·할인율 기능이 실시간으로 반영이 안 된다고 여러 번 "안되는데?"를 반복하는 사이클이 있었습니다.

**더 나은 방법**
증상을 구체적으로 묘사하면 AI가 훨씬 빠르게 원인을 찾습니다.

> **덜 효과적**: "할인율 올려도 금액이 안 바뀌어"
> **더 효과적**: "할인율 10% 올리면 상단 연간총액은 바뀌는데, 아래 성능/유지/위탁 각각 금액 입력란은 숫자가 그대로야"

어느 부분은 되고 어느 부분은 안 되는지 **범위를 구체적으로** 알려주면 디버깅 시간이 크게 줄어듭니다.

---

### ⚠️ 2-4. 스크린샷을 더 적극적으로 활용할 수 있었다

이 세션에서 스크린샷을 몇 번 첨부했고, 그때 AI가 UI 문제를 즉시 파악했습니다. 텍스트로 설명하기 어려운 화면 문제는 **캡처 한 장이 긴 설명보다 훨씬 정확**합니다.

> 특히 레이아웃 문제, 버튼 위치, 숫자 표시 오류 등은 반드시 스크린샷 첨부 권장

---

### ⚠️ 2-5. PR 방향을 사전에 확인하지 않았다

**무슨 일이 있었나**
PR을 생성했을 때 브랜치 방향(어느 브랜치 → 어느 브랜치)이 의도와 달랐습니다. 스크린샷을 보고 "이게 맞아?"라고 확인했을 때 발견되었습니다.

**개선 방법**
PR 생성을 요청할 때 명시적으로 방향을 지정합니다.

> "feature/task3-airtable-fixes → main 방향으로 PR 만들어줘"

---

### ⚠️ 2-6. 계획(Plan) 승인 전에 더 꼼꼼히 읽기

AI가 Plan 모드에서 구현 계획을 제시하면 승인 전에 내용을 꼼꼼히 읽는 것이 중요합니다. 한 번은 원하지 않는 방식(읽기 전용 행 추가)이 포함된 계획을 거절했지만, 그냥 승인했다면 구현 후 다시 되돌리는 추가 작업이 필요했습니다.

**체크리스트**
계획 검토 시 확인할 것:
- [ ] 내가 원하는 방식과 일치하는가?
- [ ] 수정하지 말아야 할 부분을 건드리지는 않는가?
- [ ] 이해가 안 되는 항목은 질문하고 넘어가기

## 3. 핵심 인사이트 요약

### 바이브 코딩을 잘하는 것 = AI를 잘 쓰는 것

| 잘하는 사람의 습관 | 아쉬운 습관 |
|-----------------|-----------|
| "왜"를 설명한다 | 결과만 요청한다 |
| 계획 단계에서 방향을 잡는다 | 구현 후 수정을 반복한다 |
| 범위를 좁혀서 질문한다 | "안된다"고만 말한다 |
| 작게 검증하고 다음으로 넘어간다 | 기능을 쌓아두고 한꺼번에 테스트한다 |
| 증상을 구체적으로 묘사한다 | 증상을 추상적으로 전달한다 |
| 스크린샷을 적극 활용한다 | 텍스트로만 설명한다 |

---

### AI가 틀릴 수 있는 상황 (주의 포인트)

이번 세션에서 실제로 발생한 AI 실수 패턴:

| 상황 | 발생한 문제 | 예방법 |
|------|-----------|--------|
| 비슷한 파일명이 여러 개 있을 때 | 서버가 서빙하지 않는 파일(`toss_step_index.html`)을 수정 | "지금 서버가 실제로 사용하는 파일이 뭐야?" 먼저 확인 |
| 같은 변수명을 다른 곳에 추가할 때 | `const mult` 중복 선언 → SyntaxError → JS 전체 동작 안 함 | 변경 후 반드시 콘솔 에러 확인 |
| 계산 로직이 복잡할 때 | T27 셀에 배수 미적용 상태로 값이 들어감 | Excel 출력값을 직접 열어서 셀 값 확인 |

## 4. 동료에게 전하는 팁

바이브 코딩을 처음 시작하는 동료에게 드리는 조언입니다.

**1. 계획 모드를 반드시 활용하세요**
코드를 바로 작성하게 하는 것보다 "계획 먼저 세워줘"로 시작하면 방향이 틀렸을 때 수정 비용이 훨씬 줄어듭니다.

**2. 브라우저 F12 콘솔을 친해지세요**
코드 오류의 90%는 콘솔에 빨간 에러로 표시됩니다. AI에게 "지금 콘솔에 이런 에러가 떠" 라고 전달하면 원인을 바로 찾습니다.

**3. 커밋은 기능 하나 완성할 때마다 하세요**
"나중에 한꺼번에 커밋해야지"는 위험합니다. 중간에 잘못됐을 때 돌아올 지점이 없어집니다.

**4. 도메인 지식이 곧 경쟁력입니다**
AI는 코드를 쓸 수 있지만 소방설비 견적 구조는 모릅니다. 업무를 정확히 아는 사람이 바이브 코딩을 하면 결과물의 품질이 완전히 달라집니다.

**5. 한 번에 여러 기능을 요청하지 마세요**
"A도 수정하고 B도 수정하고 C도 바꿔줘"는 AI가 실수할 확률이 높아집니다. 하나씩 완성하고 확인한 뒤 다음 기능으로 넘어가는 것이 결국 더 빠릅니다.
